# Share Links and Reviewer Permissions

WizerView uses two review identifiers for two different jobs.

Creator routes use the internal review ID. The creator dashboard opens reviews at `/review-builder/[reviewId]`, where an authenticated owner can edit review structure, upload preview assets, manage versions, and save settings.

Client routes use the review share token. Shared client links should point to `/r/[shareToken]`, not to a raw review ID route. The share token is generated by Supabase in `reviews.share_token`.

Public reviewers do not need an account. With an active shared link, they can read the review and add review activity:

- pinned comments when comments are enabled
- overall feedback when comments are enabled
- final decisions when decisions are enabled

Public reviewers must not be able to change creator-owned structure:

- review title, client name, instructions, status, owner, project, or content snapshot
- versions/options
- assets or uploaded previews
- share settings

Decision status sync is handled in the database by `sync_review_status_from_decision()`, an insert trigger on `public.decisions`. This lets reviewer decisions update the parent review status without granting public update access to `public.reviews`.

PIN protection is currently a UI placeholder. It should not be treated as secure access control until a server-side verification flow is added.
